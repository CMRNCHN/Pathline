# Desktop SIP stack decision

**Status:** Locked (U0 gate for `desktop-sip-bridge`)
**Date:** 2026-07-17
**Authority:** `docs/plans/2026-07-14-desktop-sip-stt-e2e.md`, `client-native/README.md`, `docs/architecture-boundary.md`, `docs/threat-model.md`

## Context

The Tauri desktop shell must inject a working `window.__pathlineSipBridge` that implements the
frozen `NativeSipBridge` interface (`client/src/transport/SipTransport.ts`) so automated runs stop
falling back to `SimulatorTransport`. The bridge owns dial + RTP media + DTMF injection and must
expose raw inbound PCM for on-device STT.

The plan's default recommendation was Linphone, with PJSIP as the battle-tested alternative. This
document overrides that default in favor of a **pure-Rust** stack, because Pathline's defining goal
is privacy and auditability, and a plain-language owner decision prioritized "fully open,
inspectable calling code with no black-box vendor blobs" over day-one maturity.

## Decision

**Chosen stack: pure-Rust SIP/RTP — `rsiprtp` (single-crate user agent).**

`rsiprtp` is one crate that covers the whole client call path: SIP signaling (transactions +
dialogs), RTP/SRTP media, DTMF (RFC 4733 telephone-event), G.711/G.722/Opus codecs, and a jitter
buffer. It maps directly onto the `NativeSipBridge` methods and gives us full ownership of the audio
buffer needed for `onAudio(pcm, sampleRate)`.

If `rsiprtp` proves inadequate during the bridge build, the **only** sanctioned fallback is the
sibling pure-Rust combination `rsipstack` (SIP signaling, incl. TLS) + `rtp-engine` (RTP/RTCP,
SRTP, G.711/Opus, DTMF RFC 2833, `cpal` device + resampling), or `wavekat-sip` (a voice-pipeline
UA built on `rsipstack`). Staying inside the pure-Rust family keeps the "one vendor, never both"
rule intact and preserves full auditability.

**Explicitly rejected for v1 desktop:** Linphone SDK and PJSIP (both are C libraries consumed via
FFI; proven and reliable, but they add closed-to-casual-audit native blobs and heavier build/linker
requirements that conflict with the privacy-first, fully-inspectable goal). They remain reasonable
future pivots if pure-Rust maturity blocks shipping.

## Rationale

- **Auditability is the product.** 100% open Rust means every line of the calling path can be read
  and reviewed — no vendor SDK that could carry telemetry or opaque behavior. This is the strongest
  privacy-aligned property available at the library layer.
- **We own the audio.** Raw PCM flows through our own code straight into local STT; nothing is
  mediated by a third-party media engine we can't inspect.
- **Simpler, safer build.** No C toolchain / FFI / linker setup in the Tauri build; pure-Cargo
  dependencies reduce supply-chain and build-reproducibility risk.
- **Permissive licensing.** The pure-Rust crates are MIT/Apache-2.0 (verify per crate at pin time),
  vs. the GPL/commercial dual licensing of Linphone and PJSIP.
- **Single-crate fit.** `rsiprtp` covers signaling + media + DTMF in one dependency, satisfying the
  plan's "exactly one SIP stack" rule cleanly.

### Accepted trade-off

Pure-Rust SIP is younger and less battle-tested against real carriers than PJSIP/Linphone (NAT
traversal, codec-negotiation edge cases, SRTP interop). Mitigation: prove the loop against the lab
Asterisk first, keep the manual paste fallback as an escape hatch, and treat a C-SDK swap as a
known contingency (the `NativeSipBridge` interface is the isolation boundary, so swapping the
implementation later does not touch engine/UI code).

## Privacy reality check (read before assuming "traceless")

The library choice does **not** make calls untraceable. On the PSTN, whoever terminates the call —
the **SIP trunk / ITSP provider** — always sees the called number, timestamps, and duration
(`README.md` "Residual Metadata"). The privacy wins live in: (1) keeping secrets/numbers/audio on
device (already Pathline's architecture), and (2) choosing a privacy-respecting number/trunk
provider. Put privacy effort there, not into the SIP library. The lab's local Asterisk stands in
for a trunk during development.

## Prerequisites (macOS-first)

- Rust toolchain per `desktop/src-tauri/rust-toolchain.toml`.
- Add the crate(s) to `desktop/src-tauri/Cargo.toml` and pin exact versions; audit and record each
  crate's license at pin time.
- No system C SIP libraries required (this is the point of the pure-Rust choice). `cpal` (pulled in
  by `rtp-engine`, if used) handles cross-platform audio devices.
- TLS is provided via `rustls` (enabled by default in `rsipstack`); no OpenSSL system dependency.

## Lab TLS notes

- The lab Asterisk listens on SIP-over-TLS at `127.0.0.1:5061` with self-signed credentials
  generated by `./scripts/lab.sh` (see `lab/asterisk/generated/` and `docs/lab-run.md`).
- For localhost lab runs, trust the generated self-signed cert (or use an explicit insecure-lab
  flag) — scope any trust-all behavior to the lab profile only; never in production.
- First bridge milestone: send an INVITE to extension `1000` and advance the IVR with matched-step
  DTMF, then wire `onAudio` PCM.

## Layer rules (non-negotiable, from the plan)

The SIP bridge (`desktop/src-tauri/src/sip_bridge.rs`) may know dial, RTP, DTMF injection, and PCM
frames. It must NOT import `RunSession`, `runEngine`, or UI. Audio is delivered as mono float32 at
(preferably) 16 kHz to match the STT path in `docs/plans/2026-07-14-desktop-sip-stt-e2e.md`.
